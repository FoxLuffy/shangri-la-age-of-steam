# Roadmap Implementer Skill

This skill automates the end-to-end cycle of selecting, implementing, testing, and shipping features from the project's 3-lane roadmap system.

## Roadmap Structure

The project uses three roadmap files with prefixed upgrade lanes:

| File | Lane Prefix | Focus |
|---|---|---|
| `roadmap_story.md` | `S_phase` | World-building, lore, narrative systems |
| `roadmap_tech.md` | `T_phase` | Testing, CI, architecture, technical upgrades |
| `roadmap_new.md` | `N_phase` | Net-new gameplay features and systems |

Each phase contains multiple checkbox items (`- [ ]` = pending, `- [x]` = done).

## Phase Selection Strategy

When choosing the **next item to implement**, follow this priority logic:

1. **T_phases first when foundational**: If T_phase 1 (testing) or T_phase 2 (CI) have unchecked items, prioritize those — they enable quality gates for everything else.
2. **Round-robin across lanes**: After foundational tech is in place, rotate between lanes (S → T → N → S → ...) to ensure balanced progress. Track which lane was last implemented by checking the most recently checked-off items.
3. **Within a phase, complete all items**: Don't skip to the next phase in a lane until all items in the current phase are checked off.
4. **Dependencies**: If a phase item references infrastructure from another lane (e.g., N_phase 1 Airship UI depends on T_phase 4 Zustand state), implement the dependency first.

## Workflow Instructions

### 1. Sync and Branch
```
git checkout main
git pull origin main
git checkout -b feature/<lane>-<phase>-<short-name>
```
Example: `feature/t-phase1-vitest-setup`

### 2. Analyze Roadmaps
- Read all three roadmap files: `roadmap_story.md`, `roadmap_tech.md`, `roadmap_new.md`
- Identify the next unchecked item using the Phase Selection Strategy above
- If multiple items in the same phase are unchecked, implement them together in one PR (they belong to the same phase)

### 3. Test-Driven Development (TDD)
- **Write Tests First**: Create or modify tests in `backend/tests/` (pytest) or `frontend/src/__tests__/` (Vitest) that define expected behavior
- **Red**: Run tests — confirm they fail
- **Green**: Implement the minimal code to make tests pass
- **Refactor**: Clean up while keeping tests green
- **Full Suite**: Run the complete test suite (`pytest backend/tests/` and/or `npm test` in frontend) to catch regressions

### 4. Update Documentation
- Mark completed items as `[x]` in the relevant roadmap file
- Update `README.md` if the feature introduces new env vars, setup steps, or architecture changes
- Update `ROADMAP_STORY.md` (the legacy file) if applicable

### 5. Commit, Push, and PR
```
git add .
git commit -m "feat(<lane>): <phase> — <description>"
git push -u origin feature/<lane>-<phase>-<short-name>
gh pr create --base main --title "feat(<lane>): <phase> — <description>" --body "<auto-generated summary>"
```

### 6. CI Gate
- After creating the PR, check CI status: `gh pr checks <pr-number> --watch`
- **If CI passes (green)**: Merge immediately with `gh pr merge <pr-number> --merge --delete-branch`
- **If CI fails (red)**: Read the failure logs, fix the issues, amend/push, and re-check
- After merge, pull main: `git checkout main && git pull origin main`

### 7. Loop
- Return to Step 1 and select the next item
- Continue until the user stops the session or all roadmap items are complete

## Commit Message Convention

Use scoped conventional commits:
- `feat(story): S_phase 1 — world codex system`
- `feat(tech): T_phase 1 — vitest setup and component tests`
- `feat(new): N_phase 1 — airship hangar UI`

## Branch Naming Convention

`feature/<lane-letter>-phase<N>-<kebab-description>`
Examples:
- `feature/t-phase1-vitest-setup`
- `feature/s-phase1-codex-system`
- `feature/n-phase1-airship-navigation`
